The beginner Git problem

Git gets stressful when every command feels like it might touch the wrong file.

Students often know the goal: submit the assignment, push one file, recover from an error, or understand why VS Code says something is staged. The hard part is translating that goal into Git's model without making a risky guess.

Before

"I need to push this one CSS file, but I have other changes too."

Git commands feel connected to hidden state: branch, staging, commits, remotes, and files you did not mean to include.

After GitPal

"Here is what is staged. Here is the command. Here is why it is safe."

GitPal slows the workflow down just enough to explain the state and keep the next action scoped.

Answer repo state

which files are staged?

Staged right now:

  • src/styles/globals.css
  • src/components/Header.tsx

These are the files that would be included if you committed now.

Safe plan scoped workflow

push only globals.css to frontend branch

  1. Confirm you are on frontend.
  2. Run git status --short.
  3. Stage only globals.css.
  4. Commit and push the branch.

GitPal explains the impact before you run each command.

Clarify ambiguous intent

help me add globals.css to frontend branch

Quick check: do you want to:

  • switch to frontend and edit there,
  • copy the file from another branch, or
  • move an existing commit?
terminal 
$ git status --short
M README.md
A src/styles/globals.css

# GitPal explains:
# A means staged for commit.
# M means modified but not staged.
# Commit now would include globals.css only.
